Usine logicielle ?

Garanties de notre usine

  • Le code source déployé est versionné
  • Lien entre l'application déployée et le code source
  • Le code déployé passe les tests automatisés
  • Les environnements de compilation et d'execution sont auto-documentés
  • Si l'application plante, elle est automatiquement relancée
  • Si un serveur plante, l'application est migrée sur un autre serveur
  • Monitoring par application

Les outils utilisés

  • Gitlab comme serveur Git
  • Gitlab-CI comme moyen d'automatisation
  • Docker comme environnement d'éxecution
  • Kubernetes comme orchestrateur d'execution
  • Fluxcd pour faire la synchro GIT/K8S

Le workflow de l'usine

Demonstration

  • Modification d'un projet simple
  • Configuration locale vs configuration déployée
  • Sur le push : compilation + déploiement en intégration
  • Sur un tag : création d'une release
  • Action manuelle : déploiement en test
  • Plantage de l'application
  • Monitoring

Git et Gitlab

  • Historique du code
  • Source de vérité
  • Système de Merge-Request : relecture
  • Équivalent à Github

Gitlab-CI

  • Intégré à Gitlab
  • Principal outil d'automatisation
  • Éxecute des scripts dans des stages
  • Fonctionne via des runners
  • Les runners tournent dans K8S

Docker

  • Environnent d'execution indépendant la machine
  • Images = recette pour créer l'environnement
  • Tags identiques à ceux de GIT

Kubernetes

  • Orchestrateur de container Docker
  • Abstrait la notion de serveur
  • Configuration descriptive
  • Extensible

Fluxcd

  • Surveille un repo GIT
  • Met à jour le cluster dès que nécessaire
  • La vérité est dans Git
